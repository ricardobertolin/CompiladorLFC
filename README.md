# CompiladorLFC

Compiler for the LFC language — a custom Arduino-like language that targets the ATmega2560 microcontroller (Arduino Mega).

---

## Overview

The compiler translates programs written in the LFC language into ATmega2560 assembly, going through four stages:

```
Source (.txt)
    │
    ▼ Stage 1 — Lexical + Syntactic Analysis (ANTLR4)
    │   CompiladorLexer + CompiladorParser
    │
    ▼ Stage 2 — AST Construction
    │   CompiladorVisitor  →  dict-based AST
    │
    ▼ Stage 3 — Semantic Analysis
    │   CompiladorSemantico  →  type checking, scope, undeclared vars
    │
    ▼ Stage 4 — Code Generation
        AsmGenerator  →  ATmega2560 assembly (.asm)
```

---

## Requirements

- Python 3.10+
- ANTLR4 Python runtime 4.13.1

```bash
pip install antlr4-python3-runtime==4.13.1
```

---

## Usage

```bash
# Compile and print assembly to stdout
python main.py tests/blink_led.txt

# Write assembly to a file
python main.py tests/blink_led.txt -o output.asm

# Print the AST (JSON) and stop
python main.py tests/blink_led.txt --ast

# Run semantic analysis only (no code generation)
python main.py tests/blink_led.txt --no-asm
```

## Language

### Types

| Keyword  | Description              |
|----------|--------------------------|
| `int`    | 16-bit integer           |
| `float`  | 16-bit floating point    |
| `string` | String literal           |
| `bool`   | Boolean (TRUE / FALSE)   |
| `void`   | No return value          |

### Variable Declaration

```
int ledPin = 13
float voltage = 3.3
string name = "sensor"
```

### Hardware Calls

| Statement                     | Description                        |
|-------------------------------|------------------------------------|
| `PIN_MODE(pin, OUTPUT)`       | Set pin direction                  |
| `DIGITAL_WRITE(pin, val)`     | Write HIGH (1) or LOW (0) to pin   |
| `DIGITAL_READ(pin)`           | Read digital value from pin        |
| `ANALOG_WRITE(pin, val)`      | PWM output (0–255)                 |
| `ANALOG_READ(pin)`            | Read ADC channel                   |
| `DELAY(ms)`                   | Pause for ms milliseconds          |

> **Note:** Hardware call arguments must use **integer literals** (e.g. `DIGITAL_WRITE(12, 1)`), not variable names. This is a current grammar limitation — the parser's `stmt` rule does not allow bare identifiers as hardware call arguments.

### Control Flow

```
IF(condition) { ... }
IF(condition) { ... } ELSE { ... }
WHILE(condition) { ... }
DO { ... } WHILE(condition)
```

Conditions support: `==`, `!=`, `<=`, `>=`, `TRUE`, `FALSE`.

### Example Programs

**Blink a LED on pin 12:**
```
int A = 1
int B = 0
int TEMPO = 1000
PIN_MODE(12, OUTPUT)
WHILE(TRUE){
    DIGITAL_WRITE(12, 1)
    DELAY(1000)
    DIGITAL_WRITE(12, 0)
    DELAY(1000)
}
```

**IF/ELSE conditional:**
```
int ledPin = 13
int lightValue = 1
PIN_MODE(13, OUTPUT)
IF(lightValue == 1){
    DIGITAL_WRITE(13, 1)
} ELSE {
    DIGITAL_WRITE(13, 0)
}
```

## Project Structure

```
CompiladorLFC/
├── main.py                  # Entry point — runs the full compiler pipeline
├── CompiladorVisitor.py     # Stage 2: parse tree → dict-based AST
├── CompiladorSemantico.py   # Stage 3: semantic analysis (types, scopes)
├── AsmGenerator.py          # Stage 4: ATmega2560 assembly generation
├── Compilador.g4            # ANTLR4 grammar definition
├── CompiladorLexer.py       # Generated by ANTLR4 (do not edit)
├── CompiladorParser.py      # Generated by ANTLR4 (do not edit)
├── CompiladorListener.py    # Generated by ANTLR4 (do not edit)
├── tests/
│   ├── blink_led.txt        # Single LED blink
│   ├── blink_two_leds.txt   # Two LEDs alternating
│   ├── if_else.txt          # IF/ELSE branch
│   └── semantic_error.txt   # Triggers semantic error (duplicate variable)
└── CompiladorLFC Etapa 1.pdf
```

## Supported Arduino Mega Pins

The assembly generator maps digital pin numbers to AVR port/bit pairs for the ATmega2560:

| Pins  | Port  |
|-------|-------|
| 6–9   | PORTH |
| 10–13 | PORTB |
| 22–29 | PORTA |
| 30–37 | PORTC |
| 42–49 | PORTL |

---

## Semantic Checks

The semantic analyzer (`CompiladorSemantico.py`) reports errors for:

- Variable used before declaration
- Variable declared twice in the same scope
- Type mismatch in assignment (`int x = "hello"`)
- Arithmetic on non-numeric types
- Hardware call arguments with wrong type (pin must be `int`)
- Control-flow condition that is not boolean/int

---

## Known Limitations

- No support for arrays or multidimensional structures
- No user-defined functions or recursion
- No `for`, `switch`, or `foreach` control flow
- Hardware call arguments must be integer literals (grammar constraint)
- Assembly output is only calibrated for a 16 MHz clock

## Regenerating ANTLR4 Files

If you modify `Compilador.g4`, regenerate the lexer/parser with:

```bash
java -jar antlr-4.13.1-complete.jar -Dlanguage=Python3 Compilador.g4
```

This overwrites `CompiladorLexer.py`, `CompiladorParser.py`, and `CompiladorListener.py`.
